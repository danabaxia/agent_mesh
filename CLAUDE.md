# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```sh
npm test                                      # full suite (node --test, zero deps)
node --test test/delegate.test.js             # one test file
node --test --test-name-pattern="timeout"     # tests whose name matches a pattern
node ./bin/agent-mesh.js serve-a2a /path/to/proj # run the A2A stdio server for one folder
node ./bin/agent-mesh.js serve /path/to/proj  # run the compat MCP server for one folder
node ./bin/agent-mesh.js --help               # usage + env var list
AGENT_MESH_E2E=1 npm test                     # opt-in real-`claude` e2e (POSIX/macOS/Linux only — see note)
node scripts/live-a2a-check.mjs               # WINDOWS real-`claude` A2A live check (initialize/ping + multi-turn resume)
node scripts/demo-setup.mjs                   # materialize the App→Library demo into a disposable workspace
node scripts/eval-pair-setup.mjs              # materialize the reusable eval PAIR (app driver + lib peer) — stdio A2A, doctor-wired
node scripts/eval-trio-setup.mjs              # materialize the reusable eval TRIO (app + lib + docs) for peer-selection + two-hop
AGENT_MESH_E2E=1 node --test test/eval-mesh-e2e.test.js  # opt-in real-`claude` trio e2e (peer-selection + two-hop; POSIX-only)
node scripts/eval-a2a.mjs --list             # A2A behavior eval: scenario catalog
node scripts/eval-a2a.mjs --trials 3         # full scorecard run (REAL `claude`, ~15-30 min)
node scripts/eval-a2a.mjs --scenario 01-should-delegate --trials 1   # one cheap probe run
node scripts/eval-perf.mjs --list            # L4 perf benchmark: routing/efficiency/quality cells
node scripts/eval-perf.mjs --trials 5        # full PerfCard run (REAL `claude`, judge-scored)
node scripts/eval-adversarial.mjs --list     # L3 security battery: one scenario per invariant (I1–I7)
node scripts/eval-adversarial.mjs --min-pass-rate 1.0   # security run (REAL `claude`; any fail = regression)
```

Requires Node >= 20. No build step, no lint config, no dependencies — `package.json` has only `node --test`. The default suite is hermetic: it stubs the `claude` binary via `createFakeClaude` (a generated `.mjs` pointed at by `AGENT_MESH_CLAUDE`; defined locally in [test/delegate.test.js](test/delegate.test.js)). [test/demo-e2e.test.js](test/demo-e2e.test.js) is **skipped unless `AGENT_MESH_E2E=1`** — it spawns a real `claude` in folder A that delegates a real write to folder B, the regression net for wire framing / do-mode write permission / cross-folder confinement that the stubbed suites cannot see.

**Real-`claude` e2e is platform-split — do not be confused when it "skips" on Windows:**
- [test/demo-e2e.test.js](test/demo-e2e.test.js) is **POSIX-only (macOS/Linux)**. It detects and spawns the CLI with `execFileSync('claude', ...)`, which on Windows cannot resolve `claude` (the launcher is a `.cmd`/`.ps1` shim; `execFile` won't append `.cmd` or run a batch file without a shell). So on win32 every test in it skips with a `POSIX-only e2e …` reason **even when `claude` is installed** — that is expected, not a failure.
- On **Windows**, the equivalent live check is [scripts/live-a2a-check.mjs](scripts/live-a2a-check.mjs): a real-`claude` run over the live `serve-a2a` JSON-RPC wire (`initialize`/`ping` + a two-turn ask proving multi-turn session resume), using the Windows-aware spawn path ([src/process.js](src/process.js) `resolveSpawnTarget`, which follows the `.cmd` npm shim to the real `.exe`). Keep the two in sync when the wire/protocol changes.

The A2A behavior eval (scripts/eval-a2a.mjs + eval/) is a REAL-claude scorecard, not a test gate — see docs/superpowers/specs/2026-06-10-a2a-behavior-evals-design.md; its harness is hermetically tested in test/eval-harness.test.js.

The **performance benchmark** (scripts/eval-perf.mjs + eval/perf/, L4) is a REAL-claude **PerfCard** scorecard (also not a gate): per routing cell (3/6/12 peers × disjoint/confusable) it meters routing accuracy (from delegation *edges*, not answer text), efficiency (tokens/$ from run-record `usage`, latency from the Task `agentmesh/metrics` block), and judge-scored answer quality, reported as one composite, efficiency-normalized card (quality-per-token / wasted-hops) so over-delegation can't win. Spec: docs/superpowers/specs/2026-06-13-mesh-perf-benchmark-design.md; hermetic coverage in test/perf-harness.test.js. (Note: `normalizeMetrics` in src/a2a/protocol.js whitelists only latency fields, so cost-capture's token fields ride the **run record**, not the Task metrics — the efficiency meter reads cost from run records accordingly.)

The **adversarial battery** (scripts/eval-adversarial.mjs + eval/adversarial/, L3) is a REAL-claude security scorecard with **one hard-gated scenario per invariant** (I1 AGENT.md-as-data · I2 anti-spoof metadata · I3 single-root write · I4 recursion cycle · I5 reserved-env override · I6 ask-only bridge · I7 registry marker). Reuses the behavior-eval runner/scorecard; the recommended security run is `--min-pass-rate 1.0` (any failure = security regression). New `eval/probes.mjs` gates (`noExternalWrite`/`refusedWith`/`envNotOverridden`/`peerNotLoaded`/`noUnexpectedDelegation`) + `buildMesh`'s `rawRegistry` escape hatch (plant unmarked/env-overriding registries). Spec: docs/superpowers/specs/2026-06-13-adversarial-eval-battery-design.md; hermetic coverage (probes + the real readManagedRegistry enforcement + scenario validity) in test/adversarial-harness.test.js. It's the integration pipeline's L3 job (was skip-guarded; now lights up).

**Two CI pipelines (see the [evaluation methodology](docs/superpowers/specs/2026-06-13-evaluation-methodology-design.md), L0–L4):** `.github/workflows/ci.yml` is the **per-PR L0 hermetic gate** — every PR into `main` gets the full linux+windows × node 20/22 matrix (`run-all-tests.mjs`, stubbed `claude`); it is the only merge blocker. To conserve Actions minutes (private repo: Windows bills 2×), a `matrix` job picks the cell set by event: `pull_request`/`workflow_dispatch` → full cross-platform matrix; `push` → Linux-only smoke, and only on the trunk `main` — feature-branch pushes don't run CI (their PR is the gate). Trunk-based development: `main` is the single living line; the retired `vN-development` lines are gone from the triggers. `.github/workflows/integration.yml` is a **separate nightly pipeline** that tracks **`main`** (`INTEGRATION_REF: main`) and runs the integration tier against a **real `claude`** (`CLAUDE_CODE_OAUTH_TOKEN` secret): L1 e2e (gates the nightly) + L2 behavior eval (record-only scorecard) + L3 adversarial (gates when present) + L4 perf (record-only) — the L3/L4 scripts have landed, so those tiers now run live. It **never gates per-PR merges**. Its shape is lint-tested in test/integration-workflow.test.js; spec: docs/superpowers/specs/2026-06-13-integration-test-pipeline-design.md.

## What this is

`agent-mesh serve-a2a <folder>` is the reference A2A stdio server fronting exactly **one** project folder. `agent-mesh serve-a2a-http <folder>` is the HTTP/JSON-RPC binding of the same contract (standard A2A interop path). A caller wires peers through a static registry, sends `SendMessage` (A2A v1.0), and receives an A2A `Task`. `agent-mesh serve <folder>` remains as the MCP compatibility server from the validated baseline. There is **no central registry, broker, or dispatcher** by design. Read `PROJECT.md` for the full PRD, security rationale, and the pinned wire contract; it is the source of truth and must stay consistent with the code.

## Architecture

Deliberate split: a **pure core** (unit-provable safety logic) plus a **thin impure shell** (only the `claude`/`git` spawns touch the outside world).

- [bin/agent-mesh.js](bin/agent-mesh.js) → [src/cli.js](src/cli.js): `realpath`-canonicalize the folder, then start either `createA2AStdioServer` (`serve-a2a`) or `createMcpServer` (`serve`).
- [src/a2a/protocol.js](src/a2a/protocol.js): A2A `AgentCard`/`Message` validation, delegate-result → `Task` mapping, namespaced `agentmesh/*` metadata, and JSON-RPC helpers.
- [src/a2a/stdio-server.js](src/a2a/stdio-server.js): newline-delimited JSON-RPC over stdio. Implements `initialize`/`ping`/`SendMessage` (A2A v1.0; Task wrapped as the `task` oneof member); valid bad inputs become rejected `Task` data, while protocol failures remain JSON-RPC errors.
- [src/a2a/stdio-client.js](src/a2a/stdio-client.js) + [src/a2a/registry.js](src/a2a/registry.js): static registry normalization, long-lived peer process spawn, and `client.send(peer, message)`.
- [src/mcp.js](src/mcp.js): compatibility MCP server. Implements only `initialize`/`ping`/`tools/list`/`tools/call`. Every `delegate_task` goes through a per-server `SerialQueue` ([src/lock.js](src/lock.js)) — at most one in-flight `claude -p` per folder, keeping the git snapshot race-free.
- [src/delegate.js](src/delegate.js): the spawn pipeline — validate input, enter call context, capture pre-run change state, spawn `claude -p`, recompute changed files, build the result, write the run log. `ask` → read-only tool allowlist; `do` → read tools + structured write tools and a generated temp `--settings` file that registers the PreToolUse path-guard hook. When the agent has a marked `registry.json` with peers, it also injects the **framework peer bridge** as the worker's onward-delegation MCP server (reserved name `agentmesh_peerbridge`) so the worker can delegate to its peers. The worker runs with `--output-format json`; `parseResultEnvelope` reads the terminal envelope for the `summary` (`.result`) **and** a `usage` block (tokens, `total_cost_usd`, `num_turns`, `api_ms`) captured into the run record + the A2A `metrics` block, falling back to the raw text tail (`usage: null`) when the output isn't a parseable envelope (timeout/error). Pure observability — no model-facing surface change. Spec: [docs/superpowers/specs/2026-06-13-delegate-cost-capture-design.md](docs/superpowers/specs/2026-06-13-delegate-cost-capture-design.md).
- [src/a2a/peer-bridge.js](src/a2a/peer-bridge.js): the **onward-delegation** surface (PROJECT.md §1.6 carve-out). One framework MCP server with generic verbs `list_peers` / `delegate_to_peer({peer,mode,task})` — the peer is an argument (data), never a per-peer tool. `delegate_to_peer` does a real A2A `SendMessage` via `createA2AClient` to a peer from the **marker-validated** `registry.json` (`readManagedRegistry` in [src/a2a/registry.js](src/a2a/registry.js)). **ask-only in v1** (refuses non-`ask` with `mode_disabled` before any spawn); the reserved bridge env (`AGENT_MESH_MODE`/`MESH_ROOT`/`MESH_CEILING` + threaded path/depth) cannot be overridden by registry `peer.env`. Spec: [docs/superpowers/specs/2026-06-06-onward-delegation-design.md](docs/superpowers/specs/2026-06-06-onward-delegation-design.md).
- [src/board/*](src/board/) + [hooks/board-notify.js](hooks/board-notify.js): the **mesh task board** — A assigns a durable task to a peer B (`create_task_for_peer` on the `agentmesh_peerbridge` bridge; ask-safe, no `claude -p` spawn), B picks it up in its OWN interactive session via the `board-notify` SessionStart hook and advances it (`list_my_tasks`/`update_my_task`) through `assigned → acknowledged → in-progress → done`, and A is notified once on completion. `task-state.js` is the pure lifecycle (only the `to` agent may advance; `from`/`to`/`id` are framework-stamped); `store.js` is the atomic file-per-task store under `<mesh-root>/mesh/board/tasks/`; `identity.js` resolves mesh-root + self-name. Wired per peered agent by `doctor` (bridge `.mcp.json` mesh-env stamp + the SessionStart hook). Spec: [docs/superpowers/specs/2026-06-15-mesh-task-handoff-design.md](docs/superpowers/specs/2026-06-15-mesh-task-handoff-design.md).
- [src/context.js](src/context.js): recursion guard. Call-path + remaining depth are read **only from this process's own env** (`AGENT_MESH_PATH` = newline-separated canonical paths, `AGENT_MESH_DEPTH` = int) and re-threaded into the child's env. A cycle (own root already in path) or exhausted depth → `refused`.
- [src/path-guard.js](src/path-guard.js) + [hooks/path-guard.js](hooks/path-guard.js): the write boundary. The hook reads the tool call from stdin, extracts the explicit path arg, `realpath`-canonicalizes it (resolving symlinks, preserving missing nested segments), and `exit(2)` (deny) if it is not under `AGENT_MESH_ROOT`.
- [src/change-detect.js](src/change-detect.js): `files_changed` = git porcelain delta **∪** files whose SHA-256 content digest changed (so re-touching an already-dirty file is still caught). Non-git folder → `files_changed: null`. Sets `preexisting_dirty` / `best_effort` flags.
- [src/description.js](src/description.js): reads `AGENT.md` **once at server start**, whitespace-collapsed and length-bounded (`MAX_DESCRIPTION_CHARS`), to build each peer's `delegate_task` tool description. Never re-read per task.
- [src/mesh-health/core.js](src/mesh-health/core.js) + [src/mesh-health/server.js](src/mesh-health/server.js): read-only health verbs for the mesh-manager agent — `check_conformance` (doctor/conformance dry-run as data), `ping_agent` (real A2A initialize/ping probe, manifest-validated name, tree-kill timeout), `triage_logs` (run-log + schedule-state failure scan). Served by the hidden CLI verb `serve-mesh-health [mesh-root]` and registered per-mesh in `<mesh-root>/mesh/mcp.json` with the `x-agentmesh readOnly` marker (ask-mode grant). Spec: [docs/superpowers/specs/2026-06-11-mesh-manager-agent-design.md](docs/superpowers/specs/2026-06-11-mesh-manager-agent-design.md).
- [src/dashboard/health-model.js](src/dashboard/health-model.js) + [src/dashboard/health-collect.js](src/dashboard/health-collect.js) + [src/dashboard/public/health-view.js](src/dashboard/public/health-view.js): the **Health "Vital Signs" view** — a dedicated top-level dashboard view answering "is every mesh organ alive, with no dead mechanism?" using **passive signals only** (no probe spawns). `health-model.js` is the PURE heart: `buildHealthReport(...)` classifies five organs (Agents · Jobs & Daemon · Task Board · Pipeline & Conformance · Cognition) into per-agent liveness + an overall verdict, builds activity-history day-buckets, and `renderHealthReport()` emits a human-readable doctor's report — all from injected data, fully unit-tested. The **honesty rule**: a mechanism is reported `dead` only when something with a known cadence has demonstrably stopped (job-bearing agent silent past `agentDeadMs`, stuck/overdue job, stale daemon heart, stuck board task); an idle on-demand agent is shown `idle`, never false-dead. `health-collect.js` is the thin tolerant shell (reads run logs, schedule-state, heartbeat.json, board, activity JSONL, agent-folder byte-sizes for cognition `prompt`/memory-long-short split, daemon-log mtime — every read try/degrades). `GET /api/health` serves the full model; **backward-compatible** — the old heartbeat keys (`summary`/`findings`/`openEscalations`) stay at top level + under `organs.jobs` so the legacy Graph-view Health panel keeps working; the route never 500s. Config: `AGENT_MESH_HEALTH_*` (see Config). Spec: [docs/superpowers/specs/2026-06-21-mesh-health-vitals-view-design.md](docs/superpowers/specs/2026-06-21-mesh-health-vitals-view-design.md).
- [src/mesh-health/health-alert.js](src/mesh-health/health-alert.js): the **proactive health-alert planner** (issue #361) — PURE: given a `HealthReport` + the prior sweep's state it returns `{ opens, closes, state }`, opening a deduped `needs-human` issue when an organ transitions to CRITICAL and closing it on recovery (re-opens if it goes critical again). One issue per organ; the rendered doctor's report is attached. Inherits health-model's honesty rule (it only reacts to what the model already classified critical). Wired by the dev-society daemon's health-alert tick (collectHealth + `gh`); the passive Vital Signs view is unchanged — this supplements it. Tests: [test/health-alert.test.js](test/health-alert.test.js).
- [src/session-provenance.js](src/session-provenance.js) + [src/dashboard/public/follow-policy.js](src/dashboard/public/follow-policy.js) / [timeline-model.js](src/dashboard/public/timeline-model.js) / [src/dashboard/resume-command.js](src/dashboard/resume-command.js): review-first session management. delegateTask tags every framework spawn with a `worker:<route>` create event (shared provenance store), so the dashboard deterministically auto-follows the USER's live CLI session (origins cli/dashboard/headroom; never peer:*/worker:*) and stitches sessions into one canvas timeline. The UI never spawns terminals — it copies exact `claude --resume <id>` commands (`/open-terminal` is deprecated). Spec: [docs/superpowers/specs/2026-06-13-review-first-session-management-design.md](docs/superpowers/specs/2026-06-13-review-first-session-management-design.md).
- [src/contract.js](src/contract.js) / [src/errors.js](src/errors.js): input validation and the result/error shapes.

## Lessons learned

- **Dashboard terminal buttons must reopen exact sessions.** The 2026-06-09 bug: every other click of the Claude CLI terminal button appeared to alternate between correct and wrong context. Root cause was using `claude --continue` whenever the canonical session was "newest"; `--continue` is a recency heuristic, so another terminal or transcript touch in the same cwd can make Claude choose a different context. Aha moment: if the UI has a concrete session id, always pass that UUID through as `--resume <id>`; use `--session-id <id>` only for a reserved canonical id whose transcript does not exist yet. Do not use `--continue` from dashboard/session launch routes. Regression coverage lives in `test/shell-endpoint.test.js` and `test/session-routes.test.js`.
- **Windows `claude` spawn: never spawn a `.cmd`/`.bat` without unwrapping it first.** Modern Node refuses to spawn a `.cmd`/`.bat` without a shell (CVE-2024-27980) → `spawn EINVAL`. `src/process.js` `resolveSpawnTarget` follows the npm `.cmd` shim to its wrapped `.exe`/`.js` (directly spawnable) — for BOTH a bare `claude` (PATH search) AND a concrete path like `AGENT_MESH_CLAUDE=C:\…\claude.cmd` (the 2026-06-09 `resolveConcreteWin` fix; before it, only `.js` was special-cased so a concrete `.cmd` hit EINVAL). If you add a new spawn site, route it through `spawnFile`/`resolveSpawnTarget`, never a raw `spawn('claude', …)`. Regression: `test/process.test.js` (win32-guarded).
- **Real-`claude` e2e is platform-split.** `test/demo-e2e.test.js` is POSIX-only and *skips* on Windows by design (its `execFileSync('claude')` can't resolve the `.cmd` shim) — that skip is not a failure. The Windows live check is `node scripts/live-a2a-check.mjs` (real A2A wire + multi-turn resume). See the Commands section note. Don't "fix" the Windows skip by weakening the POSIX test; keep the two harnesses in sync instead.
- **MCP tools race the first model turn in headless `claude -p` — never depend on first-turn tool enumeration.** Diagnosed 2026-06-10 from eval scenario 06's flake: the CLI's stream-json init event reports every MCP server as `"pending"` — even an instant no-op server — so the first model call MAY begin before `mcp__agentmesh_peerbridge__*` registers; a worker asked to "call delegate_to_peer" or to enumerate its tools early then truthfully denies having them, while a functional ask ("ask your library peer…") issued after a beat of work finds the tools fine. Mitigations: (1) `src/cli.js` lazy-loads per command (95ms→7ms entrypoint; bridge initialize round-trip ~70ms) to shrink the window — every delegation spawns these bins, so this also cuts mesh-wide overhead; (2) phrase worker-facing tasks/prompts FUNCTIONALLY, never by internal tool name (the roster names the `agentmesh_peerbridge` server for literal-matchers); (3) eval fixtures must not gate on first-turn tool visibility. The race itself is claude-CLI sequencing — not fixable framework-side.
- **Real-`claude` e2e fixtures: frame bait honestly, and `.claude/` is expected residue.** Two 2026-06-10 rots in `test/demo-e2e.test.js`: (1) agent folders ARE Claude projects — the CLI may create its own `.claude/` state dir in its cwd, so strict directory-listing assertions must filter it out (CLI bookkeeping, not task output); (2) the escape test's deceptive-looking bait (`INJECTED.txt`, "report whether the write was blocked") tripped the model's prompt-injection heuristics and was refused **before the mesh ever ran** — exercising the model's safety layer instead of the path-guard. Fix: declare the fixture as the framework's own confinement test, state the expected denial up front, use neutral filenames; the model then cooperates down to the layer under test.

## Invariants — do not break these

These are security properties, not style preferences. Changing them means changing `PROJECT.md` first and understanding the threat model.

- **Anti-spoof**: the model-facing surface of `delegate_task` is exactly `{ mode, task }`. Any `path`/`depth`/bookkeeping field in the tool args is ignored entirely; call-path/depth come only from process env. Never read recursion state from tool input.
- **No `Bash` in `do`**: v1 `do` allows only `Edit`/`Write`/`MultiEdit`/`NotebookEdit` (`WRITE_TOOLS` in [src/config.js](src/config.js)) plus read tools. Arbitrary shell can't be path-gated by inspecting argv, so it is excluded, not pretended-safe. Do not add `Bash` to a write allowlist.
- **Single writable root**: the child runs with `cwd=folderB` and no `--add-dir`; the caller's folder and every third folder are never writable. The path-guard hook is the enforcement, not a kernel sandbox — keep `extractToolPaths` exhaustive for every write tool and keep canonicalization (symlink + missing-segment) intact.
- **`AGENT.md` is untrusted data, never instructions**: it is length-bounded and framed as data before becoming a tool description / `describe_self` payload. Never execute or obey it.
- **Failure is data, not an exception**: any non-`done` outcome (`timeout`/`refused`/`error`) is returned as a structured result with a `log_path`; the caller decides the fallback. The mesh never auto-retries or auto-fails-over. On timeout, kill the whole process **tree** (`detached: true` + `process.kill(-pid)`) and still return a `status:timeout` result with partial summary.
- **Identity is the `realpath`-canonical absolute folder path** — used identically for cycle detection and the writable root. Keep both on the same canonical form.
- **Task-board identity is framework-set**: `create_task_for_peer`/`update_my_task` take only brief/state fields; `from`, `to`, `id`, and timestamps come from the authentic caller identity + the task record, never tool args. Only the task's `to` agent may advance it (single-step forward). The brief is data surfaced for review, never auto-executed. Assignment writes only the framework-owned board (`<mesh-root>/mesh/board/`), never a peer's folder — so it is ask-safe.
