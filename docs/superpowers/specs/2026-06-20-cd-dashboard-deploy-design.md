# CD for the Dashboard — bring the dashboard under deploy-sync Design

**Date:** 2026-06-20
**Status:** Approved (design); pending spec review
**Topic:** Auto-update the locally-running dashboard when code is pushed to `main`, by extending the existing `deploy-sync` CD pipeline to also manage the dashboard as a launchd service.

## Problem

When code merges to `main`, the host's running services should pick it up
automatically. A pull-based CD already does this for the **dev-society daemon**:
`deploy-sync` (`com.danabaxia.agent-mesh.deploy-sync`, launchd, every 300s) runs
from a dedicated deploy checkout at `~/.agent-mesh/deploy`, hard-resets it to
`origin/main`, and `launchctl kickstart -k`s the daemon when main advances. The
logs confirm it works (`advanced … restarted` on several pushes today).

The **dashboard** is the gap. It is launched *manually* from the developer
checkout (`~/Documents/dev/agent_mesh`, `node ./bin/agent-mesh.js dashboard
dev-mesh`). That checkout is shared with a concurrent agent and is usually on a
feature branch, **not** `main` — so the dashboard a) doesn't auto-update on push,
and b) often serves stale, off-`main` code. It is the one running service outside
CD.

A GitHub Action cannot fix this: Actions run in GitHub's cloud and cannot reach a
Mac on a home network, so the deploy step must run **on the host**. The host
already has the right mechanism (the `deploy-sync` pull agent); we extend it
rather than add a new one.

## Goal

The dashboard becomes a managed launchd service that:
1. runs from the **deploy checkout** (`~/.agent-mesh/deploy`) — always `main`, never the dev branch,
2. serves the **deploy checkout's** mesh (`~/.agent-mesh/deploy/dev-mesh`) — the same mesh the deployed daemon operates on, so the dashboard shows the "production" view,
3. is **restarted by `deploy-sync` on advance**, alongside the daemon,
4. starts on login and survives crashes.

Latency is the existing ~5-min poll interval. Same port (7077); the dashboard
token persists across restarts, so a bookmarked tokened URL keeps working.

## Non-Goals (YAGNI)

- **No webhook / push trigger** (no inbound port/tunnel). Pull-only.
- **No interval change.** Tightening 300s → 60s is a one-line plist edit done
  separately if desired; not part of this change.
- **No health-gate or rollback for the dashboard.** It is a read-only viewer;
  `KeepAlive` + the app's single-instance reaping are sufficient. (The daemon's
  existing build-lock deferral still applies and is reused unchanged.)
- **No change to the pure `planDeploySync` logic.** Reset/restart/defer semantics
  are untouched; we only add a second restart *target* and a plist.
- **No self-hosted Actions runner** (the rejected Option B).

## Architecture

Two additions to the existing deploy stack, plus installer wiring:

```
push → main ──(CI gate, already exists)──▶ deploy-sync tick (≤300s, launchd)
   git fetch + reset --hard origin/main          [unchanged]
   if advanced (and not build-busy):
       kickstart daemon      (required)          [unchanged]
       kickstart dashboard   (best-effort)       [NEW]
   ↓
   dashboard launchd service (NEW) reloads main code, serves ~/.agent-mesh/deploy/dev-mesh
   single-instance reaping → exactly one dashboard on :7077
```

The dashboard restart is **best-effort and isolated**: it runs after the daemon
restart and its failure is logged but never throws, so it cannot break the
daemon path or block the persisted-state advance. (Consequence: on the rare
dashboard-kick failure — e.g. plist not loaded — the dashboard catches up on the
*next* advance rather than retrying every tick. Acceptable for a read-only
viewer; the daemon keeps its strict retry guarantee. This asymmetry is
deliberate: coupling the dashboard into the state gate would let a dashboard
problem wedge the daemon into restarting every tick.)

## Components

### 1. Dashboard launchd plist — `com.danabaxia.agent-mesh.dashboard`

Generated by the installer (see §3). Shape (mirrors the daemon/sync plists):

```xml
<plist version="1.0"><dict>
  <key>Label</key><string>com.danabaxia.agent-mesh.dashboard</string>
  <key>ProgramArguments</key><array>
    <string>$NODE_BIN</string>
    <string>$DEPLOY_ROOT/bin/agent-mesh.js</string>
    <string>dashboard</string>
    <string>$DEPLOY_ROOT/dev-mesh</string>
    <string>--no-open</string>
    <string>--port</string><string>7077</string>
  </array>
  <key>WorkingDirectory</key><string>$DEPLOY_ROOT</string>
  <key>EnvironmentVariables</key><dict>
    <key>PATH</key><string>…</string>
    <key>HOME</key><string>$HOME</string>
  </dict>
  <key>RunAtLoad</key><true/>
  <key>KeepAlive</key><true/>
  <key>StandardOutPath</key><string>$DEPLOY_ROOT/.dev-society/dashboard.out.log</string>
  <key>StandardErrorPath</key><string>$DEPLOY_ROOT/.dev-society/dashboard.err.log</string>
</dict></plist>
```

`$DEPLOY_ROOT` = `~/.agent-mesh/deploy`, `$NODE_BIN` = resolved node path (the
installer already computes both for the daemon plist).

### 2. `deploy-sync.mjs` — restart both labels

`src/dev-society/deploy-sync.js` (pure `planDeploySync`) is **unchanged** — it
still returns one `restart` boolean. The change is entirely in the **injected
restart function** at the CLI layer of `scripts/dev-society-deploy-sync.mjs`:

Add a multi-label restart helper (testable — `kick` is injectable):

```js
// kicks `required` labels (throw on failure → retried next tick) then `optional`
// labels (best-effort: failure logged, swallowed). Returns when required succeed.
export function makeMultiRestart({ required = [], optional = [], kick = launchctlKick, log = () => {} }) {
  return async () => {
    for (const label of required) await kick(label);            // throws → no state advance, retried
    for (const label of optional) {
      try { await kick(label); }
      catch (e) { log({ action: 'dashboard_restart_failed', label, error: e?.message || String(e) }); }
    }
  };
}
function launchctlKick(label) {
  return sh('launchctl', ['kickstart', '-k', `gui/${process.getuid()}/${label}`], { maxBuffer: 1 << 20 });
}
```

CLI wiring (replaces `restart: makeLaunchctlRestart(label)`):

```js
const daemonLabel = process.env.DEV_SOCIETY_DAEMON_LABEL || 'com.danabaxia.agent-mesh.dev-society';
const dashLabel = process.env.DEV_SOCIETY_DASHBOARD_LABEL || '';   // optional; empty → daemon-only (back-compat)
const restart = makeMultiRestart({
  required: [daemonLabel],
  optional: dashLabel ? [dashLabel] : [],
  log: (r) => { /* append to the same deploy-sync.log */ },
});
```

`makeLaunchctlRestart` may stay for back-compat or be removed if unused.
Back-compat: with `DEV_SOCIETY_DASHBOARD_LABEL` unset, behavior is identical to
today (daemon-only).

### 3. `dev-society-deploy-install.sh` — generate + load the dashboard plist

Add to the installer, mirroring the existing `daemon_plist`/`sync_plist`
pattern:

- `DASH_LABEL="com.danabaxia.agent-mesh.dashboard"`.
- A `dashboard_plist()` heredoc emitting the §1 plist.
- Add `<key>DEV_SOCIETY_DASHBOARD_LABEL</key><string>$DASH_LABEL</string>` to
  `sync_plist()`'s `EnvironmentVariables` (so deploy-sync learns the label).
- **Dry-run** branch: echo `[dry-run] write … $DASH_LABEL.plist`, the plist body,
  and `[dry-run] reload dashboard: …`.
- **Live** branch: `dashboard_plist > "$LA_DIR/$DASH_LABEL.plist"` then
  `reload "$DASH_LABEL" "$LA_DIR/$DASH_LABEL.plist"`.
- Update the final summary line to mention the dashboard.

## Data Flow

`push → main` → CI (existing gate) → `deploy-sync` tick → `fetch` + `reset --hard
origin/main` → `planDeploySync` says advanced → `restart()` kicks daemon
(required) then dashboard (best-effort) → both processes exec the new
`bin/agent-mesh.js` from `~/.agent-mesh/deploy` → dashboard's single-instance
reaping ensures one server on :7077 serving `~/.agent-mesh/deploy/dev-mesh`.

## Error Handling

- **Dashboard kick failure** → logged (`dashboard_restart_failed`), swallowed;
  daemon restart + state advance proceed. Catches up next advance.
- **Dashboard crash** → `KeepAlive` restarts it (launchd).
- **Build in flight** → existing `buildBusy`/`deferredRestart` path defers *both*
  restarts (the composite restart isn't called); reset still runs; next tick
  retries. Reused unchanged.
- **Double dashboard** (manual + service, or launchd overlap) → the app's
  single-instance reaping kills the older PID on startup; one server on :7077.
- **Install idempotency** → `reload()` does bootout-then-bootstrap, so
  re-running the installer repoints cleanly (same as daemon/sync today).

## Testing (hermetic, `node --test`, matching existing files)

**`test/deploy-sync.test.js`** (extends; injects fakes — no launchd):
- `makeMultiRestart` kicks **all** labels in order (required then optional) —
  inject a recording `kick`, assert call order `[daemon, dashboard]`.
- A failing **optional** kick is swallowed and logged; the returned promise still
  resolves (so `writeState` runs / state advances). Inject a `kick` that throws
  for the dashboard label; assert it resolved and `log` saw
  `dashboard_restart_failed`.
- A failing **required** kick rejects (so `runDeploySyncOnce` does NOT advance
  state) — assert the promise rejects and `writeState` is not called.

**`test/deploy-install-lint.test.js`** (extends; parses the script / dry-run
output — no launchd):
- Dry-run output includes a `com.danabaxia.agent-mesh.dashboard.plist` write with
  ProgramArguments containing `dashboard`, `$DEPLOY_ROOT/dev-mesh`, `--no-open`,
  and `--port 7077`, plus `RunAtLoad` and `KeepAlive`.
- `sync_plist` output contains `DEV_SOCIETY_DASHBOARD_LABEL` =
  `com.danabaxia.agent-mesh.dashboard`.
- Dry-run includes a `reload dashboard` line (enable + kickstart), matching the
  existing assertions for daemon/sync.

## Verification (manual, on the host — after install)

1. Re-run the deploy installer (or `--dry-run` first) and confirm
   `~/Library/LaunchAgents/com.danabaxia.agent-mesh.dashboard.plist` exists and
   the job is loaded (`launchctl list | grep dashboard`).
2. Open `http://127.0.0.1:7077/?t=<token>` → dashboard serves the deploy mesh.
3. Push a trivial commit to `main`; within ~5 min the deploy-sync log shows
   `advanced … restarted`, and the served `graph-view.js` (or any changed asset)
   reflects the new commit (curl with the session cookie, or reload).
4. Confirm the dev-checkout manual dashboard is retired (one server on :7077,
   owned by launchd).

## Migration note

After this lands and the installer runs, stop launching the dashboard by hand
from the dev checkout — the launchd service owns :7077. The preview/dev dashboards
were ad-hoc; the managed service is the single source of truth.
